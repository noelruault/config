# claudio: two Claude accounts, one machine

`claudio` runs Claude Code under two isolated HOMEs (`--1` personal, `--2` work).
Defined in `aliases/utils/external`.

Each account authenticates with its own **long-lived OAuth token** generated by
`claude setup-token`, stored 0600 in that account's home and passed to claude via
`$CLAUDE_CODE_OAUTH_TOKEN` at launch. This is auth precedence #5 (above `/login`
OAuth), so claude uses the token directly and never reads the macOS Keychain.

## Why tokens, not the Keychain

The macOS Keychain stores Claude's OAuth creds under one fixed slot (service
`Claude Code-credentials`, account `$USER`) that **every** HOME collides on, no
matter what `CLAUDE_CONFIG_DIR`/`HOME` is set to (anthropics/claude-code#20553).
The old claudio swapped one account's creds into that single slot before each
launch and saved them back on exit. That raced badly: run `--1` and `--2` at the
same time and claude's mid-session token refresh would read/overwrite the shared
slot, both sessions converged onto one account, and the save-on-exit then wrote
the wrong creds into the other account's snapshot. Result: both accounts ended up
on one login, sharing usage.

Long-lived tokens fix this by construction: they live in per-home files, are read
from the environment, don't rotate, and never touch the Keychain. Two sessions can
run concurrently with zero cross-contamination. `setup-token` requires a Pro / Max
/ Team / Enterprise plan and bills the subscription, not API credits.

## Usage

    claudio              # choose account interactively, then launch claude
    claudio --1          # personal
    claudio --2          # work
    claudio --1 ...args  # pass args through to claude
    claudio --1 login    # (re)generate + store the personal account's token
    claudio --2 login    # (re)generate + store the work account's token
    claudio login        # choose an account, then generate its token
    claudio --resume <id>           # auto-detect which home holds <id>
    claudio --1 --resume <id>       # offer to copy <id> from the other home first
    claudio -h           # help

State on disk (one token per account):

    ~/.claude-personal-home/.claude/.claudio-oauth-token   # personal token (0600)
    ~/.claude-work-home/.claude/.claudio-oauth-token        # work token (0600)

## First-time setup (and re-auth)

Run the `login` once per account. `setup-token` runs its OWN browser OAuth, so the
token belongs to **whichever account you authorize in the browser**, NOT to the
`--1`/`--2` flag. Log into the matching account each time:

1. `claudio --1 login` → in the browser, log into your **personal** account →
   authorize. claudio captures the printed token into the personal token file.
2. `claudio --2 login` → in the browser, log into your **work** account →
   authorize. Captured into the work token file.

Tip: the browser may reuse an existing claude.ai cookie and silently authorize the
wrong account. Use an incognito window, or switch account on the consent screen, so
`--1` really gets personal and `--2` really gets work. Run the two logins one at a
time.

If auto-capture misses the token, claudio echoes whatever `setup-token` printed and
prompts you to paste it; copy the `sk-ant-oat...` value and press enter.

## Token expiry

`setup-token` tokens are nominally valid ~1 year, but can be invalidated earlier
(seen in the wild). Symptom: a session starts failing auth / returns 401. Fix is
cheap: re-run the matching `claudio --1 login` / `--2 login` to mint a fresh token.

## Concurrency

`claudio --1` and `claudio --2` are safe to run **at the same time** in different
terminals. Each reads only its own token file and sets only its own process env;
there is no shared slot, owner file, or save-on-exit step to race on.

## Caveat: no in-CLI usage %

`setup-token` tokens are scoped to inference only, so `/usage` no longer shows the
session/week limit bars under claudio (the limits endpoint needs `user:profile`).
This is the accepted tradeoff for concurrency-safe isolation. Inference is
unaffected. See per-account usage on claude.ai -> Settings -> Usage, or on the
desktop via the macos-widgets "Claude Usage" tile (see the OAuth reference below).

## Migrating off the old Keychain design

The old `.claudio-credentials.json` snapshots and `~/.claude/claudio-keychain-owner`
pointer are no longer used and can be deleted. Once both accounts run on tokens, the
shared Keychain slot is vestigial too; wipe it when no Keychain-based claude session
is live (harmless if it reports "could not be found"):

    security delete-generic-password -s "Claude Code-credentials" -a "$USER"

## Reference: Claude OAuth on this machine (don't break this in a refactor)

There are **two separate credential purposes**, deliberately decoupled. Conflating
them is exactly what broke things before — keep them apart.

| Purpose | Auth | Scope | Stored at | Code |
|---|---|---|---|---|
| **Run** claude (claudio) | `setup-token` long-lived token via `$CLAUDE_CODE_OAUTH_TOKEN` | inference-only | `~/.claude-{personal,work}-home/.claude/.claudio-oauth-token` | `aliases/utils/external` (`_claudio_login` / `_claudio_run`) |
| **Read** usage % (macos-widgets "Claude Usage" tile) | full-scope OAuth grant (PKCE), self-refreshed | `user:profile` | `~/.claude-{personal,work}-home/.claude/.claude-usage-refresh-token` | `macos-widgets/publisher/lib/claude-oauth.sh` + `plugins/20-quota.sh`; seed with `publisher/claude-usage-login.sh <acct>` |

Why two: the usage endpoint needs `user:profile`, which the inference `setup-token`
lacks (→ 403). And refresh tokens ROTATE on use, so the widget can't share claude's
token — each consumer needs its OWN grant or they cross-invalidate.

### Claude auth precedence (highest wins)
1. Cloud provider (`CLAUDE_CODE_USE_BEDROCK` / `_VERTEX` / `_FOUNDRY`)
2. `ANTHROPIC_AUTH_TOKEN`
3. `ANTHROPIC_API_KEY` — an API key here **bills the API**. claudio `unset`s this and #2 before launch so it can't.
4. `apiKeyHelper`
5. `CLAUDE_CODE_OAUTH_TOKEN` — what claudio uses (subscription-billed, not API)
6. `/login` OAuth (macOS Keychain, the single shared slot)

### Endpoints / client (from the claude binary, v2.1.x)
- Usage: `GET https://api.anthropic.com/api/oauth/usage` + header `anthropic-beta: oauth-2025-04-20` → `{five_hour, seven_day, seven_day_sonnet, seven_day_opus}.{utilization, resets_at}`. Needs `user:profile`.
- Client id: `9d1c250a-e61b-44d9-88ed-5944d1962f5e` (public Claude Code client)
- Authorize (SUBSCRIPTION): `https://claude.com/cai/oauth/authorize`  ← use this
- Authorize (Console/API): `https://platform.claude.com/oauth/authorize`  ← do NOT use for managed accounts
- Token: `https://platform.claude.com/v1/oauth/token`
- Redirect: `https://platform.claude.com/oauth/code/callback`
- PKCE S256, `code=true` (paste-code; no localhost callback server needed)

### Gotchas that cost hours (don't relearn them)
- **setup-token can't read usage** (inference-only → 403 "scope requirement user:profile"). That's why the CLI `/usage` is blank under claudio, and why the widget needs its own grant.
- **The Console authorize host is blocked for managed domains.** `platform.claude.com/oauth/authorize` tries to create a Console org; webbeds.com Enterprise policy blocks it ("blocking new organization creation"). The subscription host `claude.com/cai/oauth/authorize` authorizes against the existing team instead — use it for the widget seed.
- **Refresh tokens rotate** on every use. One token = one consumer. Never point both claude and the widget at the same grant.
- **Token endpoint rate-limits (429).** Space logins a few minutes apart; auth codes are single-use, so just re-run to mint a fresh one.
- **Billing:** subscription OAuth = subscription usage, never API. Only `ANTHROPIC_API_KEY` bills the API. The widget grant is scoped to `user:profile` only, so it can't even run inference.

## Tests

`tests/test_claudio.zsh` covers token capture/storage (`_claudio_login`), token
injection at launch (`_claudio_run`, including dropping `ANTHROPIC_*`), the resume
auto-detect, and the cross-home session copy. It uses a sandboxed `$HOME` and a fake
`claude` on `PATH` — it never invokes the real binary or the Keychain.
